# Implement-Yourself Track

This directory is the hands-on workshop. Starting from a stripped-down skeleton (Makefile, `pyproject.toml`, scripts, profile markdown, empty `__init__.py` files), you'll rebuild the full hybrid AI system — Deep Research Agent + LinkedIn Writing Workflow — one slice at a time, using agentic coding to drive each slice through an automated SWE↔Tester loop.

By the time the 25 tickets in `tasks/` are done, this directory is a 1:1 functional replica of the parent project's `src/` (plus the Streamlit chat UI).

The detailed READMEs (research, writing, evals) are *generated by the workshop itself* — tasks #9, #19, and #24 produce them. This README only covers how to kick off the implementation.

---

## Prerequisites

- Python ≥ 3.12 (see `.python-version`).
- [`uv`](https://docs.astral.sh/uv/) for env + dependency management.
- GNU Make.
- An MCP-compatible harness — Claude Code is the reference target; Cursor works too.
- A Nebius API key (`NEBIUS_API_KEY`) for text LLM calls.
- An Exa API key (`EXA_API_KEY`) for real-time web research.
- A Gemini API key (`GEMINI_API_KEY`) for image generation.
- Optional: an Okahu Cloud API key (`OKAHU_API_KEY`) for observability + eval traces.

## Install

> **Open your harness inside this directory, not at the repo root.** `implement_yourself/` is a self-contained project — its own `pyproject.toml`, `Makefile`, `.claude/`, and `.mcp.json`. Launching Claude Code (or Cursor) from here scopes the agent's working directory to this folder, so it can't read, grep, or otherwise peek at the reference implementation in `../src/`. That's the whole point: the SWE agent has to actually build each slice from the ticket spec.

```bash
cd implement_yourself
# then open your harness here — e.g. `claude` from inside this directory
uv sync
cp .env.example .env   # fill in NEBIUS_API_KEY, EXA_API_KEY, and GEMINI_API_KEY
```

## How the workshop runs

The 25 tickets live under `tasks/NNN-slug.groomed.md`. Each is a fully groomed spec — Scope, Acceptance Criteria, User Stories — sized to be implemented in a single agentic-coding session.

The driver is a custom Claude Code skill, **`/implement`**, plus two specialized sub-agents:

- `software-engineer` — reads the ticket, populates the skeleton, runs `make format-fix`/`lint-fix`/`format-check`/`lint-check` + the ticket's e2e Make target, hands off to QA.
- `tester` — runs the e2e target, runs an adversarial pass with 2–3 break paths, walks every Acceptance Criterion with concrete evidence, returns PASS/FAIL.

Both files live under `.claude/agents/`. The `/implement` skill itself is at `.claude/skills/implement/SKILL.md`.

The `commit-commands` plugin is enabled in `.claude/settings.json` — the skill drives the commit through `/commit-commands:commit` after the Tester PASSES.

### Harness-agnostic variant: `/implement-universal`

`/implement` relies on Claude Code's subagent dispatch (the `Task` tool) to spawn `software-engineer` and `tester` in isolated contexts. Harnesses that don't have that primitive — Cursor, Windsurf, plain Anthropic SDK loops, generic MCP clients — can run the same workflow through **`/implement-universal`** instead.

The universal skill is functionally identical to `/implement`, but instead of dispatching subagents it instructs the model to **adopt two roles in sequence** inside one conversation:

1. **Software Engineer phase** — read `.claude/skills/implement-universal/agents/software-engineer.md` and execute it end-to-end.
2. **Tester phase** — read `.claude/skills/implement-universal/agents/tester.md` and verify as a fresh reviewer (logic tickets only).

The trade-off is weaker Tester independence (the Tester phase sees the SWE phase's reasoning in the shared context window). If you're running inside Claude Code, prefer `/implement` for the cleaner subagent isolation.

### One ticket per `/implement` invocation

`/implement` is intentionally single-shot — it ships exactly one ticket per call, then stops. The workshop cadence is: invoke → narrate the diff to the audience → optionally amend → invoke again on the next ticket.

### Branch strategy

The skill creates a feature branch only on the **first** ticket (when you're starting from `main`). Every subsequent ticket detects the existing feature branch and stacks its commit on top. End state: one long-lived branch carrying all 24 commits.

If you'd prefer a different branch name than `implementing/from-scratch` (which is the auto-derived name from the first ticket), check out your own branch before invoking `/implement` for the first time:

```bash
git checkout -b implementing/from-my-idea
/implement 1
```

The skill will see you're already off `main` and reuse your branch from the start.

## Kicking off

From inside `implement_yourself/`, in Claude Code (or any harness that reads `.claude/`):

```
/implement 1
```

If your harness doesn't support Claude Code's subagent dispatch (Cursor, Windsurf, plain Anthropic SDK loops, etc.), substitute `/implement-universal` for `/implement` everywhere below — the argument forms are identical.

Variants:

| Argument | Behaviour |
|---|---|
| `/implement 1` (or `001`) | Pick the ticket whose number is `001`. |
| `/implement next` | Auto-pick the lowest-numbered ticket not yet under `tasks/done/`. |
| `/implement bootstrap-research-mcp-server` | Pick by slug. |
| `/implement tasks/003-implement-analyze-youtube-video.groomed.md` | Pick by full path. |
| `/implement` (no arg) | Skill asks "Which task?" |
| `/implement-universal …` | Same arguments as above, but runs the SWE↔Tester loop in a single conversation context (for harnesses without subagent dispatch). |

The skill then:

1. Resolves the ticket and shows you the title + scope summary.
2. Creates `implementing/from-scratch` if you're on `main` (otherwise reuses the current branch — every ticket stacks its commit on the same long-lived branch).
3. Launches the SWE agent. SWE implements + runs lint + runs the e2e Make target, hands off.
4. Launches the Tester agent. Tester verifies + runs adversarial break paths + walks every AC.
5. On Tester FAIL, re-launches the SWE with concrete feedback. Caps at 3 retries.
6. On Tester PASS, flips the ticket's `Status: pending` → `Status: done`, `git mv`s the file to `tasks/done/`, then runs `/commit-commands:commit` to land a commit on the current branch.
7. Reports a summary block: branch, files changed, commit short-SHA, AC evidence list — and ends the session.

You then review the commit (`git show HEAD`), explain it to the workshop audience, and run `/implement next` to pick up the next ticket.

If the loop stalls (3 Tester FAILs in a row without convergence), `/implement` stops and surfaces `USER ACTION REQUIRED` with the failing reports. That's your cue to pair on the implementation manually before retrying.

## Reference: useful Make targets

- `make format-fix && make lint-fix` — auto-fix formatting + linting (the agents do this).
- `make format-check && make lint-check` — verify cleanliness (agents gate on this).
- `make run-research-server` / `make run-writing-server` — boot one of the MCP servers via stdio (used by the harness; agents kill after a few seconds when verifying boot).
- `make run-ui` — boot the Streamlit chat UI (task #25) that drives both MCP servers without a harness.
- `make test-research-workflow` / `make test-writing-workflow` / `make test-end-to-end` — e2e smoke tests against the dataset's seed sample. Tickets reference these as their verification targets.
- `make eval-dev` / `make eval-test` / `make eval-online` — LLM-judge evaluations (tasks #22–#23).
- `make eval-dev` / `make eval-test` / `make eval-online` — local LLM-judge F1 evals with optional Okahu/Monocle traces.

Once the workshop is done, `src/research/README.md`, `src/writing/README.md`, and `src/writing/evals/README.md` will document each subsystem in detail — those READMEs are produced by tasks #9, #19, and #24 respectively.

## Troubleshooting

- **`/implement` doesn't appear after install** → run `/reload-plugins` in Claude Code, or restart the harness.
- **Skill complains about a missing dependency ticket** → it's advisory; you can confirm to proceed out of order, but most tickets genuinely build on each other.
- **3 Tester FAILs in a row** → don't fight the loop. Read the surfaced report, fix manually, then re-invoke.
- **Want a clean restart** → delete the populated source files, `git checkout main`, `git branch -D implementing/from-scratch` (or whatever your custom branch is), and start over from `/implement 1`.
